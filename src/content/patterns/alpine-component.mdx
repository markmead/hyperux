# Alpine.js {frontmatter.title}

`alpinejs-component` lets you extract any HyperUX pattern into a reusable component. Define the pattern markup once inside a `<template>`, then stamp it out with different configuration by passing data through the host element's `x-data`.

<LastUpdated date={frontmatter.modDate} />

## Setup

Add the CDN script to your document `<head>` **before** Alpine.js:

```html
<script defer src="https://unpkg.com/alpinejs-component@latest/dist/component.min.js"></script>
<script defer src="https://unpkg.com/alpinejs@latest/dist/cdn.min.js"></script>
```

For npm-based projects, register the plugin before `Alpine.start()`:

```js
import Alpine from 'alpinejs'
import AlpineComponent from 'alpinejs-component'

Alpine.plugin(AlpineComponent)
Alpine.start()
```

## How It Works

The data contract is the host element's `x-data`. Whatever you put there is accessible inside the template, including as arguments to HyperUX pattern constructors:

```html
<!-- Host passes configuration through x-data -->
<div
  x-data="{ triggerId: 'account-menu', triggerLabel: 'Account', menuItems: [...] }"
  x-component="'hux-dropdown'"
  x-component-styles="global"
></div>

<!-- Template reads from x-data to configure huxDropdown -->
<template id="hux-dropdown">
  <div x-data="huxDropdown({ triggerId: triggerId, menuItems: menuItems })">
    ...
  </div>
</template>
```

Adding `x-data` inside the template creates a child Alpine scope. The host's `x-data` values remain accessible through Alpine's scope chain, so `triggerLabel` (from the host) and `isOpen` (from `huxDropdown`) are both available in the template markup.

## Quick Start

### 1. Start with the pattern markup

Take the standard `huxDropdown` markup from the [Dropdown documentation](/patterns/dropdown):

```html
<div
  x-data="huxDropdown({
    triggerId: 'options-menu',
    menuItems: [
      { label: 'Profile', href: '/profile' },
      { label: 'Settings', href: '/settings' }
    ]
  })"
  class="relative inline-block"
  x-on:click.outside="closeMenu()"
>
  <button
    x-ref="dropdownTrigger"
    type="button"
    x-bind:id="triggerButtonId"
    x-bind:aria-expanded="isOpen.toString()"
    aria-haspopup="menu"
    x-bind:aria-controls="menuId"
    x-on:click="toggleMenu()"
  >
    Options
  </button>
  ...
</div>
```

### 2. Wrap it in a `<template>`

Move the markup into a `<template id="...">` element and replace any hardcoded configuration values with variables. Those variable names become the data contract that callers must fulfill:

```html
<template id="hux-dropdown">
  <div
    x-data="huxDropdown({ triggerId: triggerId, menuItems: menuItems })"
    class="relative inline-block"
    x-on:click.outside="closeMenu()"
  >
    <button
      x-ref="dropdownTrigger"
      type="button"
      x-bind:id="triggerButtonId"
      x-bind:aria-expanded="isOpen.toString()"
      aria-haspopup="menu"
      x-bind:aria-controls="menuId"
      x-on:click="toggleMenu()"
      x-text="triggerLabel"
    ></button>
    ...
  </div>
</template>
```

### 3. Use it with `x-component`

Stamp the component onto the page as many times as needed. Each instance gets its own isolated Alpine scope and independent pattern state:

```html
<div
  x-data="{
    triggerId: 'account-menu',
    triggerLabel: 'Account',
    menuItems: [
      { label: 'Profile', href: '/profile' },
      { label: 'Settings', href: '/settings' },
      { label: 'Sign out', action: 'sign-out' }
    ]
  }"
  x-component="'hux-dropdown'"
  x-component-styles="global"
></div>

<div
  x-data="{
    triggerId: 'resources-menu',
    triggerLabel: 'Resources',
    menuItems: [
      { label: 'Documentation', href: '/' },
      { label: 'GitHub', href: 'https://github.com/markmead/hyperux', target: '_blank' }
    ]
  }"
  x-component="'hux-dropdown'"
  x-component-styles="global"
></div>
```

Both dropdowns share the same `hux-dropdown` template. Changing the template — adding an icon, updating a class, modifying keyboard handling — updates every instance at once.

## Common Usage Patterns

### URL Template: Reusable `huxCopy` Button

For components you want to share across multiple pages, store the template in a plain `.html` file and load it with `x-component.url`. Place the file in your `public/` directory so it is served as a static asset.

**`/public/components/hux-copy-btn.html`:**

```html
<button
  type="button"
  x-data="huxCopy({ sourceNames: [copySource] })"
  x-on:click="copyToClipboard()"
  class="rounded-md border border-gray-300 bg-white px-4 py-2 text-sm font-medium hover:bg-gray-100"
>
  <span aria-live="polite" x-text="hasCopied ? 'Copied!' : copyLabel"></span>
</button>
```

`copySource` and `copyLabel` are resolved from the host element's `x-data`. `hasCopied` comes from `huxCopy`'s return value, available because it is a child scope of the host:

```html
<div data-hux-copy="example-snippet">...</div>

<div
  x-data="{ copySource: 'example-snippet', copyLabel: 'Copy snippet' }"
  x-component.url="'/components/hux-copy-btn.html'"
  x-component-styles="global"
></div>
```

Use the same URL template for a second copy target without duplicating markup:

```html
<div data-hux-copy="another-snippet">...</div>

<div
  x-data="{ copySource: 'another-snippet', copyLabel: 'Copy example' }"
  x-component.url="'/components/hux-copy-btn.html'"
  x-component-styles="global"
></div>
```

### `x-for` Loop of Instances

When you need many instances of the same pattern from a data array, combine `x-for` with `x-component`:

```html
<div x-data="{ menus: [...] }">
  <template x-for="menu in menus" x-bind:key="menu.triggerId">
    <div
      x-data="{ triggerId: menu.triggerId, triggerLabel: menu.triggerLabel, menuItems: menu.items }"
      x-component="'hux-dropdown'"
      x-component-styles="global"
    ></div>
  </template>
</div>
```

## Behavior Contract

- The host element's `x-data` is the component's data contract. Any value set on the host is accessible inside the template.
- `x-data` inside the template creates a nested Alpine scope. The host's data is still reachable through Alpine scope inheritance.
- Components render into Shadow DOM. Each instance gets an isolated Alpine scope, so open/close state, focus position, and any other per-instance state do not bleed between instances.
- `x-component-styles="global"` adopts all local document stylesheets (including compiled Tailwind) into the Shadow DOM so utility classes work inside the component. This is required for any HyperUX pattern that uses Tailwind classes.
- `x-component.url` fetches and caches the HTML file by URL. The cached template is reused for all instances that reference the same URL.

## Notes

- HyperUX patterns that use `this.$el.querySelectorAll` to find child elements (such as `huxTabs` focus helpers) will not find elements inside a sibling or parent Shadow DOM. Keep `huxTabs` usage with `x-for` on the light DOM or test Shadow DOM boundary behavior before componentizing those patterns.
- `document.querySelector('[data-hux-copy="..."]')` works from inside Shadow DOM because it queries the full document, not the shadow tree. `huxCopy` copy sources placed in the light DOM are found correctly.
- `alpinejs-component` is a package by [markmead](https://github.com/markmead/alpinejs-component) — the same author as HyperUX.
