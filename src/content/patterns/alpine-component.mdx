# Alpine.js {frontmatter.title}

`alpinejs-component` lets you extract any HyperUX pattern into a reusable component. Define the pattern markup once inside a `<template>`, then stamp it out with different configuration by passing data through the host element's `x-data`.

<LastUpdated date={frontmatter.modDate} />

## Setup

Add the CDN script to your document `<head>` **before** Alpine.js:

```html
<script
  defer
  src="https://cdn.jsdelivr.net/npm/alpinejs-component@2.0/dist/component.min.js"
></script>
<script defer src="https://cdn.jsdelivr.net/npm/alpinejs@3.15/dist/cdn.min.js"></script>
```

For npm-based projects, register the plugin before `Alpine.start()`:

```js
import Alpine from 'alpinejs'
import AlpineComponent from 'alpinejs-component'

Alpine.plugin(AlpineComponent)
Alpine.start()
```

## How It Works

The data contract is the host element's `x-data`. Whatever you put there is accessible inside the template, including as arguments to HyperUX pattern constructors:

```html
<!-- Host passes configuration through x-data -->
<div
  x-data="{ dropdownConfig: { triggerId: 'account-menu', triggerLabel: 'Account', menuItems: [...] } }"
  x-component="'hux-dropdown'"
  x-component-styles="global"
></div>

<!-- Template reads from x-data to configure huxDropdown -->
<template id="hux-dropdown">
  <div x-data="huxDropdown(dropdownConfig)">
    <button x-text="dropdownConfig.triggerLabel">...</button>
    ...
  </div>
</template>
```

Adding `x-data` inside the template creates a child Alpine scope. The host's `x-data` values remain accessible through Alpine's scope chain, so `dropdownConfig.triggerLabel` (from the host) and `isOpen` (from `huxDropdown`) are both available in the template markup.

## Quick Start

### 1. Start with the pattern markup

Take the standard `huxDropdown` markup from the [Dropdown documentation](/patterns/dropdown):

```html
<div
  x-data="huxDropdown({
    triggerId: 'options-menu',
    menuItems: [
      { label: 'Profile', href: '/profile' },
      { label: 'Settings', href: '/settings' }
    ]
  })"
  class="relative inline-block"
  x-on:click.outside="closeMenu()"
>
  <button
    x-ref="dropdownTrigger"
    type="button"
    x-bind:id="triggerButtonId"
    x-bind:aria-expanded="isOpen.toString()"
    aria-haspopup="menu"
    x-bind:aria-controls="menuId"
    x-on:click="toggleMenu()"
  >
    Options
  </button>
  ...
</div>
```

### 2. Wrap it in a `<template>`

Move the markup into a `<template id="...">` element and replace the hardcoded config with a single object variable. That object becomes the data contract callers must fulfill:

```html
<template id="hux-dropdown">
  <div
    x-data="huxDropdown(dropdownConfig)"
    class="relative inline-block"
    x-on:click.window="if (!$event.composedPath().includes($el)) closeMenu()"
  >
    <button
      x-ref="dropdownTrigger"
      type="button"
      x-bind:id="triggerButtonId"
      x-bind:aria-expanded="isOpen.toString()"
      aria-haspopup="menu"
      x-bind:aria-controls="menuId"
      x-on:click="toggleMenu()"
      x-text="dropdownConfig.triggerLabel"
    ></button>
    ...
  </div>
</template>
```

The host's `dropdownConfig` object is passed whole to `huxDropdown` as `initialConfig`. The pattern reads `triggerId` and `menuItems` from it. `triggerLabel` is not a `huxDropdown` option — it is read directly from `dropdownConfig` in the template markup through the parent Alpine scope.

`x-on:click.outside` is not reliable in Shadow DOM. Alpine's default outside check uses `event.target`, which is retargeted to the shadow host at the document level. Using `click.window` with `composedPath()` avoids this. `composedPath()` returns the full event path through shadow boundaries, so `includes($el)` correctly identifies clicks inside the shadow root.

### 3. Use it with `x-component`

Stamp the component onto the page as many times as needed. Each instance gets its own isolated Alpine scope and independent pattern state:

```html
<div
  x-data="{
    dropdownConfig: {
      triggerId: 'account-menu',
      triggerLabel: 'Account',
      menuItems: [
        { label: 'Profile', href: '/profile' },
        { label: 'Settings', href: '/settings' },
        { label: 'Sign out', action: 'sign-out' }
      ]
    }
  }"
  x-component="'hux-dropdown'"
  x-component-styles="global"
></div>

<div
  x-data="{
    dropdownConfig: {
      triggerId: 'resources-menu',
      triggerLabel: 'Resources',
      menuItems: [
        { label: 'Documentation', href: '/' },
        { label: 'GitHub', href: 'https://github.com/markmead/hyperux', target: '_blank' }
      ]
    }
  }"
  x-component="'hux-dropdown'"
  x-component-styles="global"
></div>
```

Both dropdowns share the same `hux-dropdown` template. Changing the template — adding an icon, updating a class, modifying keyboard handling — updates every instance at once.

## Common Usage Patterns

### URL Template: Reusable `huxCopy` Button

For components you want to share across multiple pages, store the template in a plain `.html` file and load it with `x-component.url`. Place the file in your `public/` directory so it is served as a static asset.

The template reads `copyConfig` for all options including `copyLabel` for the button text (defaults to `'Copy'` if omitted). `sourceNames` copies an element's HTML; `valueSourceNames` copies an input's value. All `huxCopy` options are configurable per-instance without changing the template.

**`/public/components/hux-copy-btn.html`:**

```html
<button
  type="button"
  x-data="huxCopy(copyConfig)"
  x-on:click="copyToClipboard()"
  class="rounded-md border border-gray-300 bg-white px-4 py-2 text-sm font-medium hover:bg-gray-100"
>
  <span aria-live="polite" x-text="hasCopied ? 'Copied!' : (copyConfig.copyLabel ?? 'Copy')"></span>
</button>
```

Copy an input's value — useful for share links or generated tokens:

```html
<input
  type="text"
  data-hux-copy="share-link-input"
  value="https://hyperux.dev/patterns/dropdown"
  readonly
/>

<div
  x-data="{ copyConfig: { valueSourceNames: ['share-link-input'], copyLabel: 'Copy link' } }"
  x-component.url="'/components/hux-copy-btn.html'"
  x-component-styles="global"
></div>
```

Copy a code snippet's markup — useful for developer-facing copy buttons:

```html
<div data-hux-copy="usage-snippet">
  &lt;div x-data="{ ... }" x-component="'hux-dropdown'"&gt;&lt;/div&gt;
</div>

<div
  x-data="{ copyConfig: { sourceNames: ['usage-snippet'], copyLabel: 'Copy snippet' } }"
  x-component.url="'/components/hux-copy-btn.html'"
  x-component-styles="global"
></div>
```

Both calls reuse the same `hux-copy-btn.html` file.

### `x-for` Loop of Instances

When you need many instances of the same pattern from a data array, combine `x-for` with `x-component`:

```html
<div x-data="{ menus: [...] }">
  <template x-for="menu in menus" x-bind:key="menu.triggerId">
    <div
      x-data="{ dropdownConfig: menu }"
      x-component="'hux-dropdown'"
      x-component-styles="global"
    ></div>
  </template>
</div>
```

### Limitations

Not every HyperUX pattern can be componentized with `alpinejs-component`. The table below summarises compatibility:

| Pattern             | Works?                                          | Notes                                                                                                                                                                                                                                                                                                                       |
| ------------------- | ----------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `huxDropdown`       | <TriangleAlert class="size-4 text-amber-500" /> | Data-driven, but `focusItem()` uses `document.getElementById` which cannot find elements inside a Shadow root — keyboard focus does not move when used as an `alpinejs-component`.                                                                                                                                          |
| `huxCopy`           | <CircleCheck class="size-4 text-green-600" />   | Fully data-driven. Demonstrated in this page.                                                                                                                                                                                                                                                                               |
| `huxCommandPalette` | <CircleCheck class="size-4 text-green-600" />   | Works with the `dialogIsOpen` bridge pattern. Demonstrated in this page.                                                                                                                                                                                                                                                    |
| `huxCombobox`       | <TriangleAlert class="size-4 text-amber-500" /> | Data-driven, but `x-on:click.outside` uses event target retargeting that breaks inside Shadow DOM — the list stays open when clicking away.                                                                                                                                                                                 |
| `huxResizable`      | <TriangleAlert class="size-4 text-amber-500" /> | Resize shell is data-driven, but the template currently hardcodes an iframe `src` — a small change is needed to accept `src` via config before it can be used as a reusable component.                                                                                                                                      |
| `huxTabs`           | <CircleX class="size-4 text-red-500" />         | Panel content is arbitrary static HTML in the DOM — cannot be passed via `x-data`.                                                                                                                                                                                                                                          |
| `huxDialog`         | <CircleX class="size-4 text-red-500" />         | As a standalone pattern, modal body content is arbitrary user HTML that cannot be passed via `x-data`. However, it can work as an internal primitive within a higher-order component template (like `huxCommandPalette`) where the dialog body is fixed and data-driven. `x-teleport` also does not work inside Shadow DOM. |
| `huxScrollSpy`      | <CircleX class="size-4 text-red-500" />         | The scrollable region with headings is static HTML; different instances need different heading markup, which cannot be passed via `x-data`.                                                                                                                                                                                 |

`huxTabs` and `huxScrollSpy` cannot be componentized because their variable parts are arbitrary HTML fragments (tab panel content, scrollable headings) that cannot be passed via `x-data`. `huxDialog` has the same constraint when used standalone, but it can serve as an internal primitive inside another component where the dialog body is fixed and data-driven — exactly how `huxCommandPalette` uses it. Any workaround that stores user markup as a string in the config object misrepresents the pattern API and limits what can be rendered.

### Nested Pattern Config: Command Palette

`huxCommandPalette` composes `huxDialog` and `huxCombobox`, with the nested combobox reading `paletteItems` from the outer scope. Each host can also pass a `comboboxConfig` object inside `paletteConfig` to configure the nested `huxCombobox` per-instance — demonstrating that nested component options are overridable just by extending the config object the host passes in.

**The `dialogIsOpen` bridge**

`x-on` attribute names cannot be dynamic, so the inner combobox cannot listen for `hux-dialog:${paletteId}:open` directly. Instead, the `x-data` IIFE (the immediately-invoked function expression that returns the merged component object) adds a `dialogIsOpen` bridge property and overrides `openDialog`/`closeDialog` to set it. The nested combobox uses `$watch('dialogIsOpen')` — Alpine traverses the scope chain upward to find the property, making the template work with any `paletteId`.

**Template:**

```html
<template id="hux-command-palette">
  <div
    x-data="(() => {
      const dialogComponent = huxDialog({ dialogId: paletteConfig.paletteId })
      const paletteComponent = huxCommandPalette(paletteConfig)
      return {
        ...dialogComponent,
        ...paletteComponent,
        paletteId: paletteConfig.paletteId,
        dialogIsOpen: false,
        init() {
          dialogComponent.init.call(this)
          paletteComponent.init.call(this)
        },
        openDialog() {
          window.dispatchEvent(new CustomEvent('hux-command-palette:open', { detail: { paletteId: paletteConfig.paletteId } }))
          dialogComponent.openDialog.call(this)
          this.dialogIsOpen = true
        },
        closeDialog() {
          dialogComponent.closeDialog.call(this)
          this.dialogIsOpen = false
        },
      }
    })()"
    x-on:keydown.window="handleGlobalShortcut($event)"
    x-on:hux-command-palette:open.window="if ($event.detail.paletteId !== paletteId) closeDialog()"
    x-on:hux-command-palette:close.window="closeDialog()"
  >
    ...
    <!-- Inner combobox watches dialogIsOpen from the parent scope -->
    <!-- and reads comboboxConfig from paletteConfig for per-instance overrides -->
    <div
      x-data="{
        ...huxCombobox({
          optionItems: paletteItems.map(...),
          hideOptionItemsWhenSearchQueryEmpty: !paletteConfig.comboboxConfig?.showAllItemsOnOpen
        }),
        init() {
          this.configureTemplateRefs()
          this.$watch('dialogIsOpen', (isDialogOpen) => {
            if (isDialogOpen) this.onPaletteOpen()
            else this.onPaletteClose()
          })
        },
        onPaletteOpen() { ... },
        onPaletteClose() { ... },
      }"
    >
      <!-- input uses paletteConfig.comboboxConfig?.searchPlaceholder -->
      <input
        type="text"
        x-bind:placeholder="paletteConfig.comboboxConfig?.searchPlaceholder || 'Search...'"
        ...
      />
      ...
    </div>
  </div>
</template>
```

**Two instances with different nested config:**

The `comboboxConfig` key inside `paletteConfig` is read by the template to configure the nested `huxCombobox`. Because each host sets its own `paletteConfig`, each instance gets a different combobox behaviour without any changes to the template.

```html
<!-- Docs search: type to see results (showAllItemsOnOpen: false), Cmd+P -->
<div
  x-data="{
    paletteConfig: {
      paletteId: 'nav-palette',
      shortcutKey: 'p',
      buttonLabel: 'Open Docs Search',
      paletteItems: [
        { label: 'Scroll Spy Docs', href: '/patterns/scroll-spy' },
        { label: 'Open GitHub', href: 'https://github.com/markmead/hyperux', target: '_blank' }
      ],
      comboboxConfig: {
        searchPlaceholder: 'Type to search docs...',
        showAllItemsOnOpen: false
      }
    }
  }"
  x-component="'hux-command-palette'"
  x-component-styles="global"
></div>

<!-- Actions palette: shows all on open (showAllItemsOnOpen: true), Cmd+B; host handles action -->
<div
  x-data="{
    paletteConfig: {
      paletteId: 'actions-palette',
      shortcutKey: 'b',
      buttonLabel: 'Open Actions',
      paletteItems: [
        { label: 'Dropdown Docs', href: '/patterns/dropdown' },
        { label: 'Copy page URL', action: 'copy-page-url' }
      ],
      comboboxConfig: {
        searchPlaceholder: 'Browse or filter actions...',
        showAllItemsOnOpen: true
      }
    }
  }"
  x-component="'hux-command-palette'"
  x-component-styles="global"
  x-on:hux-command-palette:actions-palette:copy-page-url.window="navigator.clipboard.writeText(globalThis.location.href).catch(() => {})"
></div>
```

Both instances share the same template. The docs-search palette hides results until the user types; the actions palette shows all items immediately on open. The `buttonLabel` makes each trigger button visually distinct. The `copy-page-url` action item handler sits on the host element in the light DOM — events dispatched with `composed: true` cross the Shadow DOM boundary so the host `x-on` listener receives them.

**Limitation:**

`x-teleport` cannot be used inside a shadow root. Elements teleported from a shadow root to `body` lose access to the shadow scope. Modal overlays must be rendered inline; `position: fixed` still anchors them to the viewport.

## Behavior Contract

- The host element's `x-data` is the component's data contract. Any value set on the host is accessible inside the template.
- `x-data` inside the template creates a nested Alpine scope. The host's data is still reachable through Alpine scope inheritance.
- Components render into Shadow DOM. Each instance gets an isolated Alpine scope, so open/close state, focus position, and any other per-instance state do not bleed between instances.
- `x-component-styles="global"` adopts all local document stylesheets (including compiled Tailwind) into the Shadow DOM so utility classes work inside the component. This is required for any HyperUX pattern that uses Tailwind classes.
- `x-component.url` fetches and caches the HTML file by URL. The cached template is reused for all instances that reference the same URL.

## Notes

- `document.querySelector('[data-hux-copy="..."]')` works from inside Shadow DOM because it queries the full document, not the shadow tree. `huxCopy` copy sources placed in the light DOM are found correctly.
- `alpinejs-component` is a package by [markmead](https://github.com/markmead/alpinejs-component) — the same author as HyperUX.
